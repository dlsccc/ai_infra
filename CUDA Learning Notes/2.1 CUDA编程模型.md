# 内存管理
**在这里添加一个内存结构图，寄存器、sram、L1/L2 cache、dram的关系**  
内存管理在传统串行程序是非常常见的，寄存器空间，栈空间内的内存由机器自己管理，堆空间由用户控制分配和释放，CUDA程序同样，CUDA提供的API可以分配管理device上的内存，也可以用CDUA管理host上的内存，host上的传统标准库也能完成主机内存管理。以下为host原有API与CUDA C的API对比：  
| 标准C函数 | CUDA C | 说明 |
| :----:| :----: | :----: |
| malloc | cudaMalloc | 内存分配 |
| memcpy | cudaMemcpy | 内存复制 |
| memset | cudaMemset | 内存设置 |
| free | cudaFree | 释放内存 |

`cudaMemcpy`内存拷贝可以完成四个过程，分别为hosttohost, hosttodevice, devicetohost, dovicetodevice，函数成功则返回cudaSuccess，否则cudaErrorMemoryAllocation，使用`char* cudaGetErrorString(cudaError_t error)`可以读取错误代码的详细信息。  
## cuda的内存层次
共享内存（shared memory）；全局内存（global memory）  
# 线程管理
一个核函数只能有一个grid，一个grid有很多block，每个block有很多个thread  
一个block中的线程可以完成同步和共享内存的协作。  
**🚨不同block内的线程不能相互影响，他们是物理隔离的。**  
通过给线程编号将其区分开，使得不同线程执行相同代码，对应不同数据。依靠`blockIdx`、`threadIdx`来确定线程编号，如下反映线程的坐标信息，这些是unit3类型，是设备端执行时可见的，不可以在核函数运行时修改。  
```
blockIdx.x
blockIdx.y
blockIdx.z
threadIdx.x
threadIdx.y
threadIdx.z
```
`blockDim`、`gridDim`是dim3类型，是手工定义的，host端可见，明确grid和block的范围。  
# 核函数
## 启动核函数
```
kernel_name<<<grid,block>>>(argument list);
```
其中`<<<grid,block>>>`是对内核的配置，可以配置内核线程的数目以及使用线程的布局。  
```
kernel_name<<<1,32>>>(argument list);    //一个block
kernel_name<<<32,1>>>(argument list);    //32个block
//这两个代码执行结果一致，但是效率会有差异
```
host启动核函数后，控制权立刻回到主机，而不是等待设备完成核函数的运行，使用`cudaError_t cudaDeviceSynchronize(void);`可以显式地使主机等待设备端执行完毕。也有隐式的方法，比如内存拷贝函数，必须要等待设备端执行完毕才能完成内存拷贝。  
**所有CUDA核函数启动都是异步的**
## 编写核函数
kernel核函数编写有一下限制：  
1. 只能访问设备内存
2. 必须有void返回类型
3. 不支持可变数量的参数
4. 不支持静态变量
5. 显示异步行为
## 验证核函数
验证核函数和错误处理，都是一些开发过程中防止出现错误的技巧，实践过程中逐渐积累。
